# 知识网络工作流 v2.5.0 — 原子/原创分离版

> **核心原则**：笔记的价值来自人的思考，不是 AI 的包装。AI 负责搬运和格式化，人负责判断和命名。

> **v2.5 变更摘要（相对 v2.4）**：① 原子笔记重新定义为「仅已被证实的知识」；② 原创内容（新术语/新思维模型/新概念）拆出为独立「原创库」；③ 已证实与原创各按术语/思维模型/概念三形态细分，每形态下再按 7 学科分；④ 命名 PREFIX 体系重构（TC/TM/TN + OT/OM/OC）；⑤ 去重与类型判断逻辑相应更新；⑥ 文末新增缩写全称对照表。

---

## 一、立场声明（AI 必须遵守）

1. **AI 是园丁，不是作者**。AI 可以整理格式、建立链接、生成图谱，但**不能代替用户判断内容的价值**。
2. **手动笔记优先**。当用户手写笔记与 AI 建议冲突时，以用户为准。
3. **拒绝景观化**。AI 生成的解码分析必须是**可操作的**——要么指向具体行动，要么揭示具体矛盾，不能是"看起来深刻"的空话。
4. **命名权属于人**。概念的名字、分类、重要性判断，必须由用户确认或授权。
5. **已证实 ≠ 原创，物理分离**（v2.5 核心）。原子笔记只放已被证实/研究出的知识；用户新创的术语、思维模型、概念一律进原创库，不得混入原子笔记。

---

## 二、origin 血统标记（轻量版）

**3 种标记**，写在编码笔记的 frontmatter 中。原子笔记与原创库笔记不需要 origin（原子笔记通过 `source_note` 追溯，原创库通过 `source_note` + `is_original: true` 标识）。解码笔记不需要 origin（默认 AI 分析，通过 `analysis_of` 追溯）。

| 标记 | 含义 | 使用场景 | 智力来源 |
|------|------|---------|---------|
| `manual` | 手写/原创 | 用户独立写的笔记，或用户口述 AI 仅做格式化 | **人** |
| `ai-collaboration` | AI 协作 | 用户在对话中与 AI 共同塑形的内容 | **人 + AI** |
| `external` | 外部资料 | 摘录的书、文章、视频（**人类作者写的**） | **外部人** |

**规则**：
- `ai-collaboration` 的编码笔记必须在 7 天内被用户 review，否则视为草稿。
- `external` 必须在 30 天内被解码，否则归档到 `05-Resources/`。
- **没有"遗产税"、没有"重写仪式"、没有比例限制**。

**AI 独立生成内容的去向**：
- AI 独立生成的内容（如 HOTSPOT 自动简报）**不进入编码笔记**，放入 `AI融合笔记/`。
- 用户从中发现价值，需**手动提取**并以 `manual` 或 `external` 重新创建编码笔记。

## 三、编码笔记规范

### 3.1 命名格式

```
[CODE]-[YYYYMMDD]-[SEQ]@[TYPE]-标题.md
```

| 组成 | 格式 | 示例 |
|------|------|------|
| CODE | 2 位大写学科码 | `DK`, `LE`, `CE`, `PA`, `AP`, `LT`, `XX` |
| DATE | 年月日（**笔记创建入库日期**） | `20250510` |
| SEQ | 当日序号（从 001 开始） | `001` |
| TYPE | 版本状态 | `v1`, `draft`, `final`, `archive` |

**示例**：`DK-20250510-001@v1-阅读《思考，快与慢》笔记.md`

**⚠️ 日期规范**：DATE 必须是**当天创建笔记时的日期**（入库日期），而非内容产生日期、书籍出版日期或事件发生日期。这确保了笔记的时间序列清晰可追溯。

### 3.2 存放位置与子目录规则

编码笔记存放于 `编码笔记/` 下对应学科目录，命名采用 `0X-CODE-中文名-EnglishName/`。默认平级存放；某学科 ≥ 20 条时可创建学科细分子目录（禁止用日期/序号做子目录名）。

### 3.3 Frontmatter

```yaml
---
code: "DK-20250510-001@v1"        # 必填
title: "阅读《思考，快与慢》笔记"  # 必填
origin: "manual"                   # 必填（manual / ai-collaboration / external）
created: 2025-05-10                # 必填，笔记创建入库日期（与文件名日期一致）
status: "种子"                     # 必填（种子/萌芽/成熟/归档）
source: "《思考，快与慢》"         # 选填
source_type: "书籍"                # 选填
domain: "心理学"                   # 选填
tags: []
---
```

### 3.4 status 状态机

| 状态 | 含义 | 进入条件（触发） |
|------|------|---------------|
| `种子` | 刚写入，尚未解码 | `@编码` 完成时自动置入 |
| `萌芽` | 已解码，派生笔记已提取且双链已建立 | `@解码` 完成 + 派生笔记生成 + 双向链接建立后，AI 自动置入 |
| `成熟` | 派生笔记之间已建立横向关联 | `@知识图谱` 确认派生笔记间有 ≥1 条横向链接，AI 自动置入 |
| `归档` | 过时或不再关注 | 仅由**用户手动**置入，AI 不得自动归档 |

**规则**：状态只能正向流转或由用户手动回退。AI 每次执行 `@解码`/`@知识图谱` 后必须检查并更新 status，并在回执中报告。

## 四、解码笔记规范

### 4.1 触发方式

用户发送 `@解码 [编码]` 或 `@解码 [一段文本]`。

### 4.2 命名格式

```
[CODE]-[YYYYMMDD]-[SEQ]@解码-标题-解码.md
```

**示例**：`DK-20250510-001@解码-阅读《思考，快与慢》分析.md`

**⚠️ 日期规范**：DATE 必须是**当天创建解码笔记时的日期**（入库日期），通常与对应编码笔记的日期一致（同日解码），但如果隔天解码则使用解码当天的日期。CODE 和 SEQ 从编码笔记继承以保持关联。

### 4.3 双向链接建立规则

解码笔记创建时必须建立完整的双向链接网络：

1. **解码笔记内**：在开头或"来源"章节链接到编码笔记
   ```markdown
   ## 来源
   
   解码自：[[DK-20250510-001@v1]]
   ```

2. **提取原子/原创笔记时**，在新笔记中建立回链：
   ```markdown
   ## 来源
   
   解码自：[[DK-20250510-001@解码-标题]]
   源自编码：[[DK-20250510-001@v1]]
   ```

3. **同步更新编码笔记**，在末尾追加"派生笔记"章节：
   ```markdown
   ## 派生笔记
   
   - [[TC-CE-概念名]] - 已证实术语
   - [[TM-DK-模型名]] - 已证实思维模型
   - [[OC-CE-概念名]] - 原创概念
   ```

4. **同步更新解码笔记**，在末尾追加"提取的知识单元"章节：
   ```markdown
   ## 提取的知识单元
   
   - [[TC-CE-概念名]]
   - [[TM-DK-模型名]]
   - [[OC-CE-概念名]]
   ```

**链接格式**：推荐使用简洁格式 `[[概念名]]` 或 `[[文件名]]`，让 Obsidian 自动匹配；当有歧义时使用 `[[文件名|显示文本]]`。**禁止**在链接中使用 `@v1` 等版本号后缀。

### 4.4 解码内容框架（7 维度 + 反景观化硬约束）

1. **核心命题提取**
2. **矛盾分析**（主要矛盾、对立统一）
3. **前提批判**（隐含假设）
4. **逻辑检验**
5. **跨学科关联**
6. **局限与延伸**
7. **实践指向**（必须可执行）

**反景观化硬约束（AI 必须遵守）**：
- 每个维度若无实质内容，**必须写"本维度无实质发现"**，禁止套话填充。
- **维度 2/3**：必须指向源文本中的**具体句子或情境**，禁止抽象空转。
- **维度 7**：必须是可在 7 天内执行的**具体动作**（含对象、条件、可验证结果）。写不出则该解码笔记降级为草稿，不进入提取流程。
- 自检：解码完成后 AI 自问"这段删掉会损失什么具体信息？"——答不出即景观化空话，删除。

### 4.5 Frontmatter

```yaml
---
analysis_of: "DK-20250510-001@v1"   # 必填，被解码的编码笔记
source_origin: "manual"              # 必填，继承编码笔记 origin
created: 2025-05-10                  # 必填，解码笔记创建入库日期
status: "种子"
tags: []
---
```

## 五、原子笔记规范（仅已被证实的知识）⭐v2.5 重构

> **定义（v2.5）**：原子笔记只存放**已被证实、已被研究出的**知识单元——包括已证实的术语、已证实的思维模型、已证实的概念。**用户新创的一律不进此处，进原创库（第六章）。**

### 5.1 三种形态与 PREFIX

| PREFIX | 形态 | 定义 | 判断标准 | 示例 |
|:---:|------|------|---------|------|
| `TC` | 已证实**术语** | 有权威来源的学术/行业术语 | 有明确英文原名 / 权威来源 / 独立词条 | 符号暴力（布迪厄） |
| `TM` | 已证实**思维模型** | 公认的、被广泛验证的模型/框架 | 学界或业界公认、有出处 | 第一性原理、SWOT |
| `TN` | 已证实**概念** | 已被研究确立的概念 | 有学术定义、非用户自创 | 认知失调 |

### 5.2 命名格式

```
[PREFIX]-[CODE]-概念名.md
```

| 组成 | 含义 | 取值 |
|------|------|------|
| PREFIX | 形态 | `TC` / `TM` / `TN` |
| CODE | 学科编码 | `CE`/`DK`/`LE`/`LT`/`AP`/`PA`/`XX` |
| 概念名 | 中文名 | 如 `符号暴力` |

**示例**：
- `TC-CE-符号暴力.md`（已证实术语）
- `TM-DK-第一性原理.md`（已证实思维模型）
- `TN-CE-认知失调.md`（已证实概念）

**禁止**：序号、日期、`@v1` 后缀。

### 5.3 目录结构（三层：形态 → 学科）

```
03-Atomic/                         # 原子笔记 = 已被证实的
├── TC-术语/
│   ├── TC-CE-认知进化/
│   ├── TC-DK-学科知识/
│   ├── TC-LE-人生体验/
│   ├── TC-LT-文学创作/
│   ├── TC-AP-艺术感知/
│   ├── TC-PA-实践活动/
│   └── TC-XX-交叉学科/
├── TM-思维模型/
│   ├── TM-CE-认知进化/
│   ├── TM-DK-学科知识/
│   ├── TM-LE-人生体验/
│   ├── TM-LT-文学创作/
│   ├── TM-AP-艺术感知/
│   ├── TM-PA-实践活动/
│   └── TM-XX-交叉学科/
├── TN-概念/
│   ├── TN-CE-认知进化/
│   ├── TN-DK-学科知识/
│   ├── TN-LE-人生体验/
│   ├── TN-LT-文学创作/
│   ├── TN-AP-艺术感知/
│   ├── TN-PA-实践活动/
│   └── TN-XX-交叉学科/
└── 视图/
```

**规则**：第一层按 PREFIX（TC/TM/TN），第二层按学科 CODE，文件存第二层下，**禁止第三层**。学科归属通过 frontmatter 的 `discipline_code` 标记。

### 5.4 Frontmatter

```yaml
---
code: "TC-CE-符号暴力"              # 必填，与文件名一致
title: "符号暴力"
form: "术语"                        # 必填（术语/思维模型/概念）
discipline_code: "CE"              # 必填（LE/DK/AP/CE/PA/LT/XX）
source_authoritative: "布迪厄《区分》"  # 必填，权威来源
source_note:                        # 必填，回链编码笔记
  - "DK-20250510-001@v1"
is_original: false                  # 原子笔记恒为 false
created: 2025-05-10
status: "种子"
tags: []
---
```

### 5.5 双向链接要求

原子笔记创建时必须在正文中建立回链到源笔记（编码笔记和解码笔记）：

```markdown
## 来源

解码自：[[DK-20250510-001@解码-标题]]  
源自编码：[[DK-20250510-001@v1]]

## 相关概念

- [[其他相关概念]] - 横向链接由 @知识图谱 阶段建立
```

**双向链接完整性检查**：`@解码` 完成后，AI 必须确认：
- ✅ 原子笔记已回链到编码/解码笔记
- ✅ 编码笔记已追加"派生笔记"章节
- ✅ 解码笔记已追加"提取的知识单元"章节
- ✅ 编码笔记 status 已更新为"萌芽"

## 六、原创库规范（用户新创的知识）⭐v2.5 新增

### 6.1 三种形态与 PREFIX

| PREFIX | 形态 | 定义 | 示例 |
|:---:|------|------|------|
| `OT` | 原创**术语** | 用户自创的新术语 | （你的新术语） |
| `OM` | 原创**思维模型** | 用户自创的方法论/框架 | AB面分析框架、话语语法还原法 |
| `OC` | 原创**概念** | 用户自创的新概念 | 框架觉知 |

### 6.2 命名格式

```
[PREFIX]-[CODE]-概念名.md
```

**示例**：
- `OT-CE-（新术语）.md`
- `OM-CE-AB面分析框架.md`

文件名可选加 `*` 标识原创，如 `OC-CE-框架觉知*.md`，但以 frontmatter `is_original: true` 为准。

## 七、提取与去重规则（v2.5 更新）

### 7.1 提取时的第一分叉：已证实 vs 原创

```
提取一个知识单元时，AI 先判断：
这是已被证实/研究出的知识，还是用户新创的？
├── 已证实 → 进 03-Atomic（原子笔记），is_original: false
│            再判形态：术语(TC) / 思维模型(TM) / 概念(TN)
└── 原创   → 进 04-原创（原创库），is_original: true
             再判形态：术语(OT) / 思维模型(OM) / 概念(OC)
```

**判断"是否原创"的标准**：有权威学术/行业来源 = 已证实；无外部来源、由用户自创 = 原创。**AI 拿不准时必须询问用户，不得擅自归类。**

### 7.2 形态判断（第二层）

```
是已有明确来源的学术/行业术语？
├── 是 → 术语（TC / OT）
└── 否 → 是可复用的方法论/操作框架？（名称含框架/模型/方法/步法，有操作步骤）
    ├── 是 → 思维模型（TM / OM）
    └── 否 → 概念（TN / OC）
```

### 7.3 去重（按概念名全库搜索，不按文件名）

- 提取概念名（去 PREFIX 和 CODE），在 `03-Atomic/` 与 `04-原创/` 下**递归全库搜索**。
- **同名 + 同库 + 同学科 + 同形态** → 合并更新，追加 `source_note`，不建新文件。
- **同名 + 不同学科** → 查注册表复用主学科，不建跨学科重复；用户明确要求才建，标 `is_cross_discipline: true`。
- **同名但一个在原子、一个在原创** → 暂停，报告用户确认（这是重要边界，可能是把已证实误判为原创或反之）。
- **禁止**自动加 `1`/`2` 后缀；冲突无法解决则暂停询问用户（选项：合并/跨学科实例/改名/跳过）。

### 7.4 概念注册表（写入时机）

保存在 `07-System/concept-registry.yaml`，记录每个概念的库归属、主学科、主版本、形态、原创性。

```yaml
符号暴力:
  library: "atomic"           # atomic（原子）/ original（原创）
  primary_discipline: "CE"
```

## 八、AI融合笔记定位

`AI融合笔记/` 存放所有**原始 AI 输出**（对话记录 + AI 独立生成物），**不进入编码/原子/原创体系**。AI 禁止自动将总结写入其他区；总结要么经用户确认进编码笔记，要么丢弃。只有经人工判断和重塑的内容才能进入正式体系。
