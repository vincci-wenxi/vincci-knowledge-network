# 文希知识网络 v2.5 完整说明与使用指南

> 面向首次公开发布的 v2.5 说明：介绍系统定位、核心优势、知识结构、脚本职责和落地步骤。
> 众推客科技 · 文希AI社区 ｜ 2026.07

---

## 一、系统定位

文希知识网络 v2.5 是一套以 Obsidian 为核心的个人知识网络工作流。它把输入、加工、沉淀、连接和输出拆成清晰的层级，让人的判断保留在关键位置，让 AI 承担整理、检查、链接和格式化工作。

核心原则：

- 人负责判断内容价值、命名概念和确认分类。
- AI 负责格式整理、双向链接、去重检查、状态流转和图谱辅助。
- 已证实知识与个人原创内容分开存放，避免来源、权威性和创造性混在一起。
- 输出必须可操作、可追溯、可检查，拒绝只显得深刻但不能落地的分析。

## 二、Skill 核心优势

### 1. 把写笔记变成可执行流程

这个 skill 不是单纯给笔记分类，而是把知识处理拆成明确动作：先收集原始材料，再 `@编码`，再 `@解码`，再提取原子/原创知识单元，最后通过双向链接和图谱维护长期网络。用户不用每次重新设计流程，只需要按节点推进。

### 2. 明确人和 AI 的分工边界

人负责判断内容价值、命名概念、确认分类和决定是否进入正式体系；AI 负责整理格式、执行检查、建立链接、维护状态和提示冲突。这样可以利用 AI 的处理能力，但不把判断权交给 AI。

### 3. 避免草稿污染正式知识库

原始想法、读书摘录、临时材料先进入 `00-Inbox`；AI 原始输出和对话记录先进入 `AI融合笔记`。只有经过 `@编码` 确认后的内容才进入 `编码笔记/`，正式笔记区因此保持干净、可追溯。

### 4. 让知识从单篇笔记变成网络

skill 要求编码笔记、解码笔记、原子/原创笔记之间建立来源链、派生链和横向链接。每个知识单元都能追溯到来源，也能与其他概念发生连接，而不是停留在孤立摘录。

### 5. 内置质量控制

工作流内置去重、注册表、幽灵链接检查、状态机、双链自检和反景观化约束。它不只追求生成更多内容，也要求内容可验证、可维护、可行动。

### 6. 面向输出而不是只做收藏

`Output/`、`MOC`、`MAP` 和状态流转把知识沉淀连接到文章、视频脚本、社群内容和商业项目。这个 skill 的目标不是堆资料，而是让知识能持续转化为表达、决策和创作。

## 三、v2.5 架构优势

### 1. 原子笔记与原创库物理分离

`03-Atomic/` 只存放已被证实或已有研究基础的知识单元；`04-原创/` 只存放用户新创的术语、思维模型和概念。这样可以同时保护知识的可靠性和原创内容的生长空间。

### 2. 三层分类结构

v2.5 采用：

```text
库归属 -> 形态 -> 学科
```

也就是先判断它是已证实知识还是原创内容，再判断它是术语、思维模型还是概念，最后归入 LE/DK/AP/CE/PA/LT/XX 七大学科。

### 3. 六种 PREFIX 清晰识别

文件名开头直接表达知识单元的归属和形态：

- `TC` / `TM` / `TN`：已证实术语、已证实思维模型、已证实概念
- `OT` / `OM` / `OC`：原创术语、原创思维模型、原创概念

### 4. 编码、解码、原子化三段式流程

- `@编码`：把手写、摘录、对话或外部资料整理成编码笔记。
- `@解码`：对编码笔记做结构化分析，提取矛盾、前提、逻辑、关联和实践指向。
- `@知识图谱`：检查派生笔记、补齐双向链接、推动状态流转并形成网络。

### 5. 注册表与去重机制

`concept-registry.yaml` 记录概念名称、PREFIX、所属库、形态、学科和路径。脚本在提取新概念前先查注册表，减少重复、误归类和跨库混淆。

### 6. 自动化维护工具

仓库提供 4 个 Python 脚本：

- `kn_common.py`：共享解析、路径、frontmatter 和扫描逻辑。
- `kn_dedup.py`：概念去重、注册表同步、跨库冲突检查。
- `kn_links.py`：双向链接检查与幽灵链接修复。
- `kn_status.py`：根据解码、派生和图谱状态推进笔记状态机。

## 四、文件架构

### 4.1 顶层全景

```text
文希知识网络/                          # Obsidian 打开这一层作为库
├── 编码笔记/                        # @编码 后生成的结构化笔记
├── 解码笔记/                        # 加工：AI 7项解码分析，镜像 7 学科
├── Obsidian Vault/                  # 核心库：PARA + 原子 + 原创 + 系统
├── Output/                          # 可发布内容
├── Business/                        # 商业项目
├── .claude/skills/                  # AI 工作流配置
└── .knowledge-network-config.yaml   # 路径配置
```

### 4.2 编码笔记：7 学科结构化记录区

```text
编码笔记/
├── 01-LE-人生体验-LifeExperience/
├── 02-DK-学科知识-DisciplineKnowledge/
├── 03-AP-艺术感知-ArtisticPerception/
├── 04-CE-认知进化-CognitiveEvolution/
├── 05-PA-实践活动-PracticalActivity/
├── 06-LT-文学创作-Literature/
└── 07-XX-交叉学科-Interdisciplinary/
```

命名格式：`CODE-YYYYMMDD-SEQ@版本-标题.md`，例如 `DK-20250510-001@v1-读书笔记.md`。`编码笔记/` 只放经过 `@编码` 确认后的结构化笔记；原始碎片先放 `00-Inbox/`，AI 原始输出先放 `AI融合笔记/`。某学科达到 20 条以上时，可以增加学科细分子目录。

### 4.3 解码笔记：镜像 7 学科加工区


### 4.5 原子笔记 vs 原创库

| 维度 | 原子笔记 `03-Atomic` | 原创库 `04-原创` |
|------|----------------------|------------------|
| 放什么 | 已被证实或已有研究基础的知识 | 用户新创的知识 |
| `is_original` | `false` | `true` |
| 术语 | `TC`，如符号暴力 | `OT` |
| 思维模型 | `TM`，如第一性原理 | `OM`，如 AB 面分析框架 |
| 概念 | `TN`，如认知失调 | `OC`，如框架觉知 |
| 命名 | `TC-CE-符号暴力.md` | `OC-CE-框架觉知.md` |

两库结构对称：形态（术语/思维模型/概念）→ 7 学科。默认保持两层结构，避免继续套第三层。

### 4.6 Output / Business

```text
Output/
├── 公众号文章/
├── 视频脚本/
└── 社群内容/

Business/            # 商业项目，按房产、电商、外贸、大健康等实际业务需要建立
```

## 五、PREFIX 速记

### 已证实知识

| PREFIX | 含义 | English |
|:---:|------|---------|
| `TC` | 已证实术语 | Trusted Term Card |
| `TM` | 已证实思维模型 | Trusted Model |
| `TN` | 已证实概念 | Trusted Concept |

### 原创内容

| PREFIX | 含义 | English |
|:---:|------|---------|
| `OT` | 原创术语 | Original Term |
| `OM` | 原创思维模型 | Original Model |
| `OC` | 原创概念 | Original Concept |

判断顺序：

1. 先判断来源：已证实还是原创。
2. 再判断形态：术语、思维模型还是概念。
3. 最后判断学科：LE/DK/AP/CE/PA/LT/XX。

AI 不确定时必须暂停并询问用户，不能擅自归类。

## 六、脚本使用指南

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

当前依赖主要是 `pyyaml`。

### 2. 创建知识库

编辑 `setup-vault.sh` 顶部的 `VAULT_ROOT`，然后运行：

```bash
bash setup-vault.sh
```

脚本会创建 v2.5 目录结构、空的 `concept-registry.yaml` 和 `.knowledge-network-config.yaml`。

### 3. 配置路径

如果手动配置，复制 `config-template.yaml`，把 `vault_root` 改成你的知识库绝对路径。完整配置如下：

```yaml
version: 2.5.0
paths:
  vault_root: "/path/to/your/vault"
  inbox: "Obsidian Vault/00-Inbox"
  atomic: "Obsidian Vault/03-Atomic"
  original: "Obsidian Vault/04-原创"
  encoded: "编码笔记"
  decoded: "解码笔记"
  output: "Output"
  resources: "Obsidian Vault/05-Resources"
  ai_fusion: "Obsidian Vault/AI融合笔记"
  registry: "Obsidian Vault/07-System/concept-registry.yaml"
  map: "Obsidian Vault/10-MAP/知识网络图"

rules:
  encoding_note_subdir_threshold: 20
  ai_collaboration_review_days: 7
  external_archive_days: 30

naming:
  atomic_note_format: "{prefix}-{code}-{concept}.md"
  original_note_format: "{prefix}-{code}-{concept}.md"
  encoding_note_format: "{code}-{date}-{seq}@{type}-{title}.md"
  decoding_note_format: "{code}-{date}-{seq}@解码-{title}-解码.md"
```

当前脚本主要读取 `paths.atomic`、`paths.original`、`paths.encoded`、`paths.decoded` 和 `paths.registry`；其他字段用于建库说明、人工核对和后续扩展。运行脚本时仍以 `--vault <ROOT>` 指定知识库根目录。

### 4. 推荐维护顺序

首次使用或集中整理时，建议按下面顺序执行，并先 dry-run：

```bash
python scripts/kn_dedup.py --vault <ROOT> scan
python scripts/kn_links.py --vault <ROOT> check
python scripts/kn_status.py --vault <ROOT> advance
```

确认报告符合预期后，再加 `--apply` 写入：

```bash
python scripts/kn_dedup.py --vault <ROOT> sync-registry --apply
python scripts/kn_links.py --vault <ROOT> fix --apply
python scripts/kn_status.py --vault <ROOT> advance --apply
```

### 5. 单个概念检查

```bash
python scripts/kn_dedup.py --vault <ROOT> check --concept 符号暴力
```

用于确认概念是否已经存在、在哪个库、是什么 PREFIX、路径在哪里。

## 七、落地步骤

1. 克隆或下载本仓库。
2. 修改 `setup-vault.sh` 中的 `VAULT_ROOT`。
3. 运行 `bash setup-vault.sh` 创建知识库。
4. 用 Obsidian 打开创建出的知识库根目录。
5. 将 `docs/SKILL-v2_5.md` 放入 AI 工具的工作流/技能配置中。
6. 先把第一条原始材料放入 `Obsidian Vault/00-Inbox/`；AI 原始输出放入 `Obsidian Vault/AI融合笔记/`。
7. 运行 `@编码`，确认后生成 `编码笔记/`。
8. 使用 `@解码` 生成解码笔记并提取知识单元。
9. 定期运行去重、链接和状态脚本维护网络质量。

## 九、使用原则

- 手动笔记优先，AI 建议不能覆盖用户判断。
- 命名权属于用户，AI 可建议但不能擅自决定重要概念名。
- 原子笔记放已证实知识，原创库放新创内容。
- 所有派生笔记必须能回链到来源。
- 每次维护先 dry-run，再 `--apply`。
- 解码分析必须落到具体文本、具体矛盾和 7 天内可执行的行动。
